# 数据获取与清洗的逻辑和准确性的改进意见更加详细一点。

以下是逐数据源的详细改进分析，直接对应 PRD §3、§9.3 和 TODO T-05～T-08 的实现层面：

***

## 数据获取与清洗逻辑准确性——详细改进意见


***

## 一、Yahoo Finance（yfinance）

**问题 1：收盘价 `available_at` 语义未定义**[^1]

PRD 的 `AvailabilityResolver` 五级优先级中，yfinance 没有"官方发布时间"可用，直接落入第3～5级（保守规则/文件检测/抓取时间）。但 yfinance 的日 OHLCV 数据实际存在三种时间点：


| 状态 | 时间 | 问题 |
| :-- | :-- | :-- |
| 盘中快照 | 交易时段内 | 收盘价未最终确认 |
| 收盘后初始价 | 16:00 ET | 调整前原始价 |
| 复权调整价 | 次日甚至更晚 | 股息/分拆后被修订 |

**改进建议**：`availability_rules.yaml` 中需新增 `yfinance_close_price` 规则，明确规定：

- `observation_time = T 日 16:00 ET（收盘时间）`
- `available_at = T 日 18:00 ET` （保守起见，等待复权因子稳定）
- 非交易日（节假日、周末）调用需由 Trading Calendar 守卫，禁止写入空行。

**问题 2：yfinance 的价格调整（Adjusted Close）是"无声修订"**[^1]

yfinance 返回的 `Close` 和 `Adj Close` 会因为股息、拆股被追溯修订，但 PRD 的 Silver 层 append-only 模型没有专门处理这类"静默替换"。如果每次抓取都把最新调整价写入，历史观测会被替换而不是追加新行，破坏 PIT 的核心假设。

**改进建议**：为 yfinance 的价格观测区分 `price_type: RAW_CLOSE | ADJ_CLOSE | SPLIT_FACTOR`，以 `(canonical_symbol, field_name, observation_time, price_type)` 作为自然主键，修订时追加新行并将旧行的 `valid_to` 标定，完全对齐 §4.2 的 append-only 模型。同时在 `quality_flags_json` 中记录 `adj_factor` 和 `split_ratio`，使特征层可以判断价格序列是否发生了追溯调整。

**问题 3：期货连续合约（GC=F, SI=F）的展期跳空未处理**[^1]

yfinance 的 `GC=F` 和 `SI=F` 是连续合约，在合约展期时价格会出现非真实的跳空（front-month 换月时价差可达数十美元）。若直接用于计算 Z-score 或收益序列，会产生极端异常值污染特征。

**改进建议**：在 T-08 特征层增加展期检测逻辑：比对 `observation_time` 附近是否存在合约切换事件（可从 CFTC 合约到期表或手动配置），若检测到展期，则在 `quality_flags_json` 中标注 `roll_adjusted: false`，并将当日收益设为 `NaN` 而不是展期价差。Z-score 窗口内如有展期跳空点需排除或使用展期调整价（Panama 法或比率法）。

## 二、FINRA Reg SHO

**问题 4：`finra_regsho_same_day` 规则与实际发布节奏不符**[^2]

TODO T-07 中 `finra_regsho_same_day` 的命名直接暗示了"当天数据当天可得"，但 FINRA Reg SHO 短卖数据的实际发布规律是：

- T 日交易数据 → T+1 营业日发布（一般在次日下午）
- 部分市场（如 NYSE Arca）在 T+2 才在 FINRA 合并数据中出现

若 `available_at = T 日`，而实际数据 T+1 才发布，则用 T 日 18:05 的 Panel 就包含了未来数据，引入约1个交易日的前视偏差。

**改进建议**：将规则改名为 `finra_regsho_t_plus_1_afternoon`，并设置：

- `available_at = observation_date + 1 个工作日 + 14:00 ET`（取保守估计）
- 在 `max_staleness` 中对应调整为 `2D`（允许跨周末时最多2日陈旧）
- 在 T-12 防泄漏测试中增加专项 case：**"T 日 18:05 的 Panel 不得包含 T 日的 FINRA Reg SHO 数据"**。

**问题 5：short volume 分母（total volume）的来源混用**[^1]

PRD 的 `flow__finra__short_ratio` 特征计算为 `short_volume / total_volume`，但 FINRA Reg SHO 的 `total volume` 字段只包含在该 ATS/Venue 上报告的成交量，并非全市场成交量（consolidated tape）。用 FINRA 局部 `total_volume` 作为分母会夸大或失真 short ratio，特别是对 GLD、IAU 这类大量在 NYSE Arca 以外成交的 ETF。

**改进建议**：在 Metric Registry 中将该字段的 `semantic_warning` 改为："short_ratio 分母为 FINRA reporting venue 成交量，非全市场 consolidated volume，不得直接与基于 SIP tape 的成交占比对比"。特征层计算时，应优先用同源 `total_volume`（保持分子分母来源一致），但在 Evidence 层的 `semantic_caveat_zh` 中明确标注这一局限，LLM 验证规则第6条需覆盖该语义约束。

## 三、CFTC COT

**问题 6：观察日（周二）与发布日（周五）的时区边界问题**[^2]

TODO T-07 的 `cftc_friday_release` 规则正确地将 `available_at` 设置为发布的周五，但缺少以下细节：

- CFTC 周度 COT 的 `observation_time = 每周二收盘时点（21:00 UTC）`
- CFTC 的发布时间为 **东部时间周五 15:30 ET**（非周五开盘），在夏令时/冬令时切换时需注意是 EDT（UTC-4）还是 EST（UTC-5）

如果 `available_at` 只存储日期而不含时刻，当 `decision_time = 周五 15:00 ET` 时会错误地认为 COT 数据已可得。

**改进建议**：

```yaml
# availability_rules.yaml 建议格式
cftc_friday_release:
  release_day_of_week: FRIDAY
  release_time_et: "15:30"
  timezone_aware: true  # 考虑夏令时
  observation_day_of_week: TUESDAY
  lag_days: 3  # 观察日到发布日
  fallback: next_business_day  # 遇节假日顺延
```

`available_at` 必须精确到 `TIMESTAMPTZ`（含时区），不能只存日期。

**问题 7：Disaggregated COT 与 Legacy COT 的字段映射混用风险**[^1]

PRD 的 `positioning_features` 中引用了 `managed_money_net`，这是 Disaggregated COT 报告（Futures + Options 合并版）的字段。但 CFTC 同时发布三种报告：Legacy、Disaggregated、TFF（金融期货）。黄金/白银使用 Disaggregated，而 VIX 期货适用 TFF 报告中的 Dealer/Asset Manager 分类，字段名称完全不同。

**改进建议**：在 Instrument Registry 中为每个标的添加 `cot_report_type: LEGACY | DISAGGREGATED | TFF`，在 CFTC Adapter 中根据该字段路由到不同解析器。`cftc_market_code` 已有（`088691` 对应黄金），但 `cot_report_type` 目前缺失，会导致解析时字段列名对不上而静默丢失数据。

## 四、FRED / ALFRED

**问题 8：`realtime_start` / `realtime_end` 参数未在 Adapter 规范中体现**[^1]

PRD 正文正确引用了 ALFRED 的 real-time period 机制，但 TODO T-05 的 `fred_alfred.py` 验收条件仅要求"Adapter 可跑通 → 落 Raw 文件"，没有验证是否正确传入了 ALFRED 专用参数：

- `realtime_start` / `realtime_end`：指定查询哪个 vintage 的历史数据
- `vintage_dates`：批量拉取特定发布时刻的数值

如果 Adapter 直接调用 FRED 普通 API（`fred/series/observations`）而非 ALFRED API（`fred/series/observations?realtime_start=...`），获取的将是**最新修订值**而非**历史发布值**，导致宏观数据回测全部使用了未来修订的数值，破坏整个 PIT 系统的核心价值。

**改进建议**：在 T-05 验收条件中增加强制项：

- `fred_alfred.py` 必须调用 ALFRED 接口，每次请求必须显式传入 `realtime_start = ingest_date`
- Raw 的 `request.json` 中必须记录 `realtime_start` 和 `realtime_end` 参数
- T-12 防泄漏测试必须新增 case：**"同一序列用 ALFRED vintage 查询 vs. FRED 最新值，两者在有修订的日期必须返回不同数值"**，若两者总是相同则说明在调用 FRED 而非 ALFRED。

**问题 9：VIX 等市场数据通过 FRED 获取的延迟被忽略**[^1]

PRD 数据源表将 VIX 列于 FRED/ALFRED 下（如 `VIXCLS` 序列），但 FRED 的 VIX 数据来自 CBOE，更新频率和延迟与 Cboe 官方直接获取不同。FRED 通常在 T+1 或 T+2 更新 `VIXCLS`，而 Cboe 官方网站的 VIX 历史数据一般在当日收盘后数小时内可得。

**改进建议**：在 Metric Registry 中为 FRED 代理的市场数据序列（`VIXCLS`、`DGS10` 等）设置独立的 `availability_rule_id = fred_market_proxy_t_plus_1`，区别于 Cboe CFE 直接获取的 VIX 期货数据，避免因发布延迟导致 Panel 使用了比实际晚1天的 VIX 数值。

## 五、Silver 层数据清洗共性问题

**问题 10：`forward_fill_allowed` 的 Silver 层实现缺乏类型标注**[^1]

PRD §8.2 的 Metric Registry 规定了 `forward_fill_allowed: true` 和 `max_staleness: 10D`，但 Silver 表（`observations_bitemporal`）的字段定义中没有 `fill_type` 或 `is_inferred` 字段。这意味着：

- 一条被前向填充的 COT 观测和一条真实观测在 Silver 表中**外观完全相同**
- `quality_status` 只有在 Feature 层才会变成 `STALE`，但血缘追溯时无法在 Silver 层区分"真实观测"和"推算值"

**改进建议**：在 `observations_bitemporal` 表中增加：

```sql
fill_type VARCHAR NOT NULL DEFAULT 'OBSERVED',
  -- 枚举: OBSERVED | FORWARD_FILLED | CALENDAR_INFERRED | INTERPOLATED
fill_source_observation_id UUID,  -- 若为填充值，指向原始观测行
fill_lag_days INTEGER             -- 填充延迟天数，用于 staleness 计算
```

**问题 11：标的映射（canonical_symbol）的歧义性**[^1]

PRD Instrument Registry 中同时存在 `GC=F`（yfinance 代码）和 `GOLD_COMEX`（canonical_symbol），但 CFTC COT 使用 `cftc_market_code = 088691`，FINRA Reg SHO 使用 ticker 如 `GLD`。多源数据在写入 Silver 时需要统一映射到 `canonical_symbol`，但 TODO T-06 的验收条件只说"标的映射"通过，没有定义**映射失败的处理策略**。

**改进建议**：在 Pandera schema 校验中增加硬性规则：

- `canonical_symbol` 必须存在于 Instrument Registry，否则拒绝写入并报 `UNMAPPED_SYMBOL` 错误
- 原始 vendor symbol（如 `088691`、`GLD`）保存在 `source_metadata_json` 中以备追溯
- 增加 T-12 case：**"未在 Registry 中注册的 symbol 不得出现在任何 Silver 或 Feature 记录中"**

## 六、ETF 申赎代理数据（P1）

**问题 12：`shares_outstanding` 与 AUM 的日期对齐陷阱**[^1]

PRD 将 ETF 的 `shares_outstanding`（如 GLD 的实物黄金持仓量代理）列为"日频"，但实际上各基金发行方（State Street、BlackRock）公布这一数据的时间并不统一——State Street 通常在 T+1 午前更新 GLD 的 `shares_outstanding`，而 BlackRock 对 IAU 的更新可能在 T 日收盘后数小时。

**改进建议**：在 T-26 的 `etf_shares.py` Adapter 中，必须从基金发行方网页抓取时记录 HTTP response 时间作为 `ingested_at`，并在 `availability_rules.yaml` 中为每个 ETF 发行方配置独立的规则：

```yaml
etf_shares_state_street:
  release_offset_hours: 22  # T日收盘后约22小时（T+1 10:00 ET）
  source: state_street_website

etf_shares_blackrock:
  release_offset_hours: 4   # T日收盘后约4小时（T日20:00 ET）
  source: blackrock_website
```

切记 `shares_outstanding` 的变化本质上是 T 日的申赎流量，不能将 T+1 发布的值当作 T 日可知。
